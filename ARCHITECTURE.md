# 🏗️ Technical Architecture (ARCHITECTURE.md)

## 📁 Directory Structure
- `/src/app/dashboard`: Core application logic, including Server Actions (`actions.js`) and main views.
- `/src/components/ui`: Presentational, reusable components (Atomic Design).
- `/src/game`: Game "Bible" (`constants.js`). Contains all static data and balancing variables.
- `/src/utils`: The heavy lifters. 
    - `gameEngine.js`: **The Brain.** Calculates "Snapshots" (real-time financial state) combining DB data with Constants to produce the "GameState".
    - `prisma.js`: Database client.
- `src/utils/economy.js`: Helper functions for financial formatting and calculations.
- `prisma/schema.prisma`: **Database Schema.** Defines the `Company` model and its fields.
- `src/app/dashboard/actions.js`: Handles all mutations (buyFacility, hireStaff, upgradeFacility, sellFacility, fireStaff). Uses strictly defined Mapping Dictionaries.
### 🖥️ UI Layer (React Components)
- `src/components/DashboardControls.js`: Orchestrates the Dashboard UI logic.
- `src/components/Inventory.js`: Visualizes the Infrastructure Tree (HQ -> Buildings).
- `src/components/ui/ActionCard.js`: Generic card for building actions (with Sell/Buy logic).
- `src/components/ui/StaffActionCard.js`: Specialized card for HR with Hire/Fire tabs and Sliders.
- `src/components/Sidebar.js` & `TopBar.js`: Layout and navigation.

## 🧱 Component Philosophy: Presentational
- **Dumb Components (UI)**: Located in `src/components/ui/`. They are **Presentational Components**. 
  - They MUST NOT contain business logic or game math.
  - They MUST receive all data via props (using `_CONF` and `_SNAP` patterns).
  - They MUST trigger actions via passed-down functions (callbacks).

## 🏷️ Naming Conventions for Data Objects
To maintain strict separation between static rules and dynamic state, we use the following suffixes:

- **`{ENTITY}_CONF`**: Used for static configuration data from `constants.js`. 
  - *Example*: `buildingData.OFFICE_CONF.name`
- **`{ENTITY}_SNAP`**: Used for calculated state data from `gameEngine.js`. 
  - *Example*: `buildingData.OFFICE_SNAP.totalMaintenance`

**Rule**: The UI should always check `_CONF` for labels/assets and `_SNAP` for current financial values.

## 🔑 Source of Truth
- **Server-Side State**: The database (Prisma) stores raw counts and levels (e.g., `officeCount: 5`).
- **Engine Snapshots**: The UI never reads raw database numbers. It consumes "Snapshots" generated by `calculateGameState`. This function is the single source of truth for all financial and capacity calculations.
- **Next.js Revalidation**: Uses `revalidatePath` to refresh the UI immediately after any Server Action (Buy/Hire/Upgrade).


## ⚙️ Core Logic Rules
1. **Snapshots**: Never use raw DB data in UI. Always pass it through `calculateGameState`.
2. **Refund Policy**: Selling buildings or firing staff returns **50%** of the `basePrice` / `costToHire`.
3. **HQ Dependency**: 
   - `HQ` level determines the total `slots` for other buildings.
   - Buildings cannot be upgraded beyond the current `HQ` capability (Roadmap).
   - The last `HQ` (count: 1) cannot be sold.
4. **Economic Squeeze**:
   - Higher building levels = Much higher Maintenance.
   - Higher building levels = Higher Staff Revenue but slightly higher Salary.

---

## 🧩 Separation of Concerns
- **Logic vs. UI**: UI components (React) are strictly for display and triggering actions. They do not calculate revenue or taxes. All math must happen in the `gameEngine.js`.
- **Atomic Actions**: Each action in `actions.js` is responsible for one specific transaction type, ensuring database integrity and consistent validation rules.
